# Meeting Ingest

Process a meeting note in `meetings/` into the synthesized layers of the brain.

## Trigger

- Manual: `/meeting-ingest <path>` or "ingest this meeting"
- Auto: invoked by `dream-cycle` for any meeting filed today that hasn't been ingested yet

## Input

A path to a meeting note in `meetings/`. The note must have frontmatter including `attendees:` (list) and `date:` (YYYY-MM-DD).

## Configuration

```yaml
vault_owner:
  full_name: "{{VAULT_OWNER_FULL_NAME}}"        # e.g., "Jane Doe"
  short_name: "{{VAULT_OWNER_SHORT_NAME}}"      # what appears in attendee lists
  aliases: []                                   # additional name variants

reserved_filename_stems:
  - claude
  - gemini
  - agents
  - readme
  - index

phase_4_stale_cutoff_days: 30
```

- **Vault owner identity:** Names matching `vault_owner.full_name`, `vault_owner.short_name`, or any entry in `vault_owner.aliases` are SKIPPED during attendee enumeration — the vault owner's identity is in [[context/user]], not `knowledge/people/`. Do not create inbox stubs for the vault owner.
- **Reserved filename stems** (case-insensitive): the stems listed in `reserved_filename_stems`. macOS and Windows filesystems are case-insensitive, so a knowledge stub at `knowledge/tools/claude.md` will be auto-loaded as `CLAUDE.md` (project instructions). When a subject's kebab-name collides with this list, prefix with the parent company or category (e.g. `claude` → `anthropic-claude`, `gemini` → `google-gemini`). Apply this guard to ALL stub filenames generated by Phases 2 and 3.
- **Phase 4 stale cutoff:** Meetings whose `date:` is more than `phase_4_stale_cutoff_days` older than today skip Phase 4 (project action candidates). Phases 2 and 3 still fire.

## Workflow

### Phase 1: Validate input

1. Read the meeting note completely.
2. Verify frontmatter has `date:` and `attendees:`. The `type:` field must be either `meeting` (post-Phase 1 shape, with optional `subtype:`) or one of the legacy meeting type values still present in historical notes: `1on1`, `l10`, `standup`, `external`, `team`, `advisory`, `client`, `client-1on1`, `coaching-call`, `external-intro`, `external-partner`, `internal-1on1`, `internal-finance`, `internal-team`, `internal-training`, `working-session`. If `attendees:` is missing, ask the user. If `type:` is none of the above, refuse.
3. Verify the file lives under `meetings/`. If not, refuse and direct the user to `RESOLVER.md`.

### Phase 2: Attendee enrichment

For each name in `attendees:`:

1. **Skip vault owner.** If the name matches the vault-owner identity (see Configuration), do nothing for this attendee — neither stub nor reciprocal link. The meeting's `attendees:` field already records their presence; their canonical identity is in [[context/user]].
2. Search `knowledge/people/` for an existing page using kebab-cased name as filename match (e.g. `Jane Doe` → `jane-doe.md`). Also fuzzy-search by reading existing people-page titles and aliases.
3. **If a page exists:**
   - **Idempotency check:** if `[[meetings/<filename>]]` already appears in the entity's `## Timeline` section, this meeting has already been ingested for this person. Skip the rest of step 3 for them.
   - Otherwise: append a timeline entry: `- <date>: <one-line summary of meeting>. ([[meetings/<filename>]])`
   - Add a typed `related:` entry: `{ link: "[[meetings/<filename>]]", type: attended_by }` (skip if already present).
   - Update `updated:` to today.
   - **Reciprocal:** add `{ link: "[[knowledge/people/<name>]]", type: attended }` to the meeting note's `related:` frontmatter (skip if already present).
4. **If no page exists:**
   - **DO NOT create directly in `knowledge/people/`.** Ambiguity (is this a real person or a misspelling?) is too risky.
   - If an inbox stub for this person already exists for today (`inbox/<today>-person-<kebab-name>.md`), append the current meeting to its `sources:` list rather than overwriting — but skip if `[[meetings/<filename>]]` is already in the stub's `sources:`.
   - Otherwise, create a new stub at `inbox/YYYY-MM-DD-person-<kebab-name>.md` with `subtype: person`, the meeting as a source, and `ingested: false`. The next manual `/knowledge-ingest` pass promotes it to `knowledge/people/`.
   - **No reciprocal link to the meeting frontmatter.** The stub's `sources:` field is the back-reference; the meeting → `knowledge/people/<name>` link is added later by `/knowledge-ingest` at promotion time. Avoiding the link now prevents broken wikilinks pointing at not-yet-existing pages.

### Phase 3: Entity mentions

Scan the meeting body for entity mentions:

1. Look for proper nouns and software/product names. Match against existing pages in `knowledge/people/`, `knowledge/companies/`, `knowledge/tools/`.
2. **If matched:**
   - **Idempotency check:** if `[[meetings/<filename>]]` already appears in the entity's `## Timeline` section, skip the rest of step 2 for this entity.
   - Otherwise: append a timeline entry to the entity page (`- <date>: Mentioned in meeting with <attendees>. ([[meetings/<filename>]])`).
   - Add `{ link: "[[meetings/<filename>]]", type: mentioned_in }` to the entity's `related:` (skip if already present).
   - Append a reciprocal `{ link: "[[knowledge/<subtype>/<name>]]", type: mentions }` to the meeting frontmatter's `related:` (skip if already present).
3. **If unmatched and the entity is a tool, product, service, or company** that is clearly a known type (e.g. "Airtable" is a tool; "Acme Corp" is a company):
   - Apply the **reserved-filename guard** (see Configuration): if `<kebab-name>.md` would collide with a reserved instruction filename, prefix with parent company or category.
   - Create a stub in the appropriate `knowledge/<subtype>/` directory with minimal frontmatter, the meeting as the source, and `tier: 3` (signals "needs enrichment").
   - Cross-link reciprocally with the meeting note (same as step 2).
4. **If unmatched and the entity is a person:**
   - Apply the body-person heuristic:
     - **Has an attached action item or commitment** (e.g. "Brief Bob on X", "Cara to redo the audit") → create an inbox stub at `inbox/YYYY-MM-DD-person-<kebab-name>.md` (same shape as Phase 2). Same dedup rule: if a stub for today already exists, append this meeting to its `sources:`.
     - **Mentioned in passing, no attached action** (e.g. "checked with Dan") → drop in unclassified mentions (Phase 6). Do not create a stub.
5. **If unmatched and unclear** (could be a person, a project, a feature, hard to tell): drop a flagged entry into the meeting note's auto-generated synthesis block (Phase 6) under "Unclassified mentions" — do not create a page.

### Phase 4: Project mentions

**Stale cutoff:** if the meeting's `date:` is more than `phase_4_stale_cutoff_days` (default 30) older than today, **skip Phase 4 entirely** for this meeting. Action candidates from old meetings have already either been completed or rotted, so flagging them on active project plans adds noise. Phases 2 and 3 (people + entity timeline edges) still fire normally — those are inherently historical observations and remain valuable for the graph.

For each `projects/<slug>/` directory, check if its name (or `state.md` title, or known aliases) appears in the meeting body. If matched:

1. Check whether `plan.md` exists.
   - **If `plan.md` exists:** read it, find or append an `## Action candidates (auto-flagged)` section at the bottom.
   - **If `plan.md` does NOT exist** but the meeting has actionable candidates for this project: create a minimal `plan.md` with frontmatter (`type: project`, today's `created:` and `updated:`, tags borrowed from `state.md`), a `# Plan` heading, and the `## Action candidates (auto-flagged)` section. Note in the synthesis block that the plan was bootstrapped.
   - **If `plan.md` does NOT exist** and the meeting has no actionable candidates for this project: skip silently.
2. Scan the meeting body for explicit decisions or action items mentioning the project. For each one, append to the action-candidates section — **but first check idempotency**: if `Source: [[meetings/<filename>]]` already appears in the section, skip all candidate appends for this meeting (this entry has been ingested before). Otherwise append:

   ```markdown
   - **<date>** — <one-line action summary>. Source: [[meetings/<filename>]]. Status: pending review.
   ```

3. **DO NOT** edit the plan body. Append to `## Action candidates (auto-flagged)` only.
4. Update the project `state.md`'s `updated:` date.

### Phase 5: Concept references

If the meeting body explicitly invokes a concept already in `knowledge/concepts/` (by name or wikilink), add a `mentioned_in` typed edge from the concept to the meeting.

### Phase 6: Write the synthesis block back into the meeting note

Append to the meeting note (NOT touching anything above):

```markdown

<!-- generated: meeting-ingest do-not-edit -->
## Ingest summary (auto-generated YYYY-MM-DD)

- Updated people pages: [[knowledge/people/...]] (+1 timeline, +1 typed edge each)
- Stubs created in inbox: [[inbox/...]]  (if any)
- Updated entity pages: [[knowledge/.../...]] (+1 timeline)
- Action candidates flagged in: [[projects/.../plan.md#action-candidates-auto-flagged]]
- Unclassified mentions (defer to user): <names>
<!-- /generated -->
```

If the synthesis block already exists from a prior ingest run, **replace** the entire block (do not append a new one).

### Phase 7: Report

Output a one-screen summary to the user:

```
Meeting ingest complete — meetings/<filename>

Attendees enriched (3):
  knowledge/people/jane-doe.md (timeline + attended_by)
  ...

Entities updated (2):
  knowledge/tools/example-tool.md (timeline + mentioned_in)
  ...

Stubs created (1):
  inbox/YYYY-MM-DD-person-bob-smith.md  ← PLEASE REVIEW
  ...

Project action candidates flagged (1):
  projects/example-project-name/plan.md  ← 2 candidates pending review

Unclassified (0)
```

## Rules

- **Never edit the user's hand-written meeting notes** above the `<!-- generated: ... -->` block. Frontmatter additions for `related:` and bumping `updated:` are the only sanctioned edits above the block.
- **Never create directly in `knowledge/people/`.** New people always go through `inbox/` first.
- **Never create files whose names collide with reserved instruction filenames** (case-insensitive — see Configuration).
- **Never edit project plan body content.** Action candidates go to a dedicated section only.
- **Existing pages get additive updates only.** Timeline appends and typed-edge additions, no rewrites.
- All typed edges follow the taxonomy in `CLAUDE.md` § "Typed `related:` frontmatter".
- All filing decisions consult `RESOLVER.md`.
